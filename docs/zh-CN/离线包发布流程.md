# 离线包发布流程

这份文档给维护者使用，用来发布 OpenDeepSeek CN 的离线包和国内镜像。普通用户只需要看《离线安装》。

## 发布原则

- 不把真实 API Key、云厂商 AK/SK、Docker 登录信息写进仓库。
- 默认脚本只在本地生成产物，不上传 OSS/COS，不推 registry。
- 推送国内镜像必须显式使用 `--push`，并设置 `OPDS_CONFIRM_PUSH=I_UNDERSTAND`。
- 发布前必须运行 release gate 和 checksum。

## 1. 准备版本号

```bash
export OPDS_VERSION=0.5.0-cn
export OPDS_IMAGE_REGISTRY=registry.cn-hangzhou.aliyuncs.com/opendeepseek
```

确认当前提交：

```bash
git rev-parse --short HEAD
```

然后更新 `release-cn.json`：

- `version`
- `commit`
- `published_at`
- 各下载地址
- 各文件 sha256
- 各镜像 digest

## 2. 先 dry-run 镜像同步

```bash
scripts/sync-images-cn.sh
```

默认只打印 docker 命令，不会推送。

确认无误后，登录国内 registry，再执行：

```bash
docker login registry.cn-hangzhou.aliyuncs.com
OPDS_CONFIRM_PUSH=I_UNDERSTAND scripts/sync-images-cn.sh --push
```

脚本会处理：

- `open-webui`
- `hermes`
- `searxng`
- `hermes-bridge`

其中 `hermes-bridge` 从本仓库 `bridge/` 构建。

## 3. 构建源码离线包

```bash
scripts/build-offline-bundle.sh --version "$OPDS_VERSION"
```

产物会写入：

```text
dist/cn/
```

包含：

- 项目源码
- `install-cn.sh`
- `install-cn.ps1`
- `docker-compose.cn.yml`
- `.env.example.cn`
- `docs/zh-CN/`
- `onboarding/`
- `checksums.txt`

## 4. 构建镜像离线包

先确保国内镜像已经在本机存在：

```bash
docker image ls | grep opendeepseek
```

然后构建：

```bash
scripts/build-offline-bundle.sh --version "$OPDS_VERSION" --with-images
```

如果本机缺少某些镜像，脚本会失败并列出缺失项。只做文档演练时可以使用：

```bash
scripts/build-offline-bundle.sh --with-images --allow-missing
```

## 5. 生成或刷新校验文件

```bash
scripts/checksums.sh -o dist/cn/checksums.txt dist/cn
```

发布前不要跳过这一步。用户侧文档会要求对照 `checksums.txt` 校验下载文件。

## 6. 上传到 OSS/COS/CDN

仓库不包含上传脚本中的密钥。维护者可以用云厂商 CLI 上传：

```bash
ossutil cp -r dist/cn oss://opendeepseek-cn/releases/v0.5.0-cn/
```

或使用腾讯云 COS CLI。上传后再更新 `release-cn.json` 的 URL 和 sha256。

## 7. 发布前验证

```bash
scripts/release-gate.sh
scripts/goal-check.sh
bash -n scripts/sync-images-cn.sh scripts/build-offline-bundle.sh scripts/checksums.sh
```

如果本机 Docker stack 已启动，正式发布前还要跑：

```bash
scripts/release-gate.sh --full
```

## 8. 回滚

如果发布后发现问题：

1. 暂停安装入口的版本指向。
2. 从 OSS/COS/CDN 下架对应离线包。
3. 保留旧版本镜像，不要直接删除用户可能仍在使用的 tag。
4. 在 README 和 release note 写清楚受影响版本与替代版本。
